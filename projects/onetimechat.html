From Flying Club Frustration to Link-Based Messaging

OneTimeChat is a messaging app that requires no download and no account. Just click a link and start sending messages right in the browser. Thirty days after the last message is sent, the conversation disappears. It's made for temporary group coordination where getting everyone to download an app is too cumbersome.

The idea came to me when I was trying to start a flying club. We had about ten people come together to start a non-profit where we would own and share an airplane together. At first, I was emailing the whole group, and for everyone to send replies meant people would have to reply-all to keep everyone in the loop. We needed a messaging app, but picking one proved difficult with a sizable group. Everyone would have to agree to download an app, create an account, and then we'd have to get everyone in the same chat together.

It would be so much easier to just email a link to a chat. And thus, the idea was born.

Clean, responsive interface that works seamlessly on desktop and mobile

No Barriers, Just Conversation

When you receive a link, you can open the chat in the browser and see the conversation immediately—no questions asked. To send messages, just set a name and an avatar icon, and you can start chatting right in the browser.

If you have an account, all messages stay tied to your identity. Change your name or icon, and it updates across all conversations you're in. To create a new conversation, you'll need an account, and you can delete any conversation you create at any time—waiting the 30 days for the conversation to go stale isn't a requirement.

Anyone who has access to the chat can share the link, and anyone with the link can view it, so there's truly no sign-in barrier. It works on both desktop and mobile devices, and if you're on desktop, you can leave the tab open and receive push notifications while the tab is in the background.

Guest users can jump right in, while account holders get persistent identity across conversations

Key Features

OneTimeChat focuses on removing friction from temporary group coordination:

Instant Access

Click a link and you're in. No app downloads, no account creation, no permission requests. Just immediate access to the conversation.

Real-Time Messaging

Messages appear instantly through WebSocket connections. Desktop users get push notifications even when the tab is in the background.

Auto-Cleanup

Conversations automatically delete 30 days after the last message, keeping your data minimal and temporary chats truly temporary.

Building for Real-Time

The app is composed of three components: a Postgres database, a Node.js backend, and a React frontend. The architecture prioritizes real-time communication while maintaining a clean separation of concerns.

Backend Architecture

The backend service is a Node.js app using the Express framework. It serves a REST API that handles all CRUD operations for conversations and messages, as well as user accounts, authentication, and authorization.

In addition to the REST API, the service delivers real-time data over WebSocket connections through Socket.IO. This dual approach means authentication and session management happen through traditional HTTP requests, while all message delivery happens over persistent WebSocket connections.

The service also runs a cron job that deletes stale conversations and messages. Postgres triggers automatically update timestamp data, keeping the deletion logic simple and reliable.

Frontend Implementation

The frontend is a React app built with TypeScript and Vite. Redux handles authentication state, while React Router manages navigation. The app handles login, signup, and session refresh through the REST API, but everything else—messaging, presence updates, profile changes—happens through the Socket.IO client.

This means all messages sent are received almost instantaneously by other members of conversations. Profile updates from logged-in users propagate in real-time to everyone in their conversations.

Messages appear instantly through WebSocket connections—no polling, no delay, no refresh required.

Learning to Deploy Beyond Managed Services

Neither the backend nor the frontend were particularly difficult to build. The backend follows standard REST API patterns and benefits from Postgres triggers for automatic timestamp updates. The frontend uses familiar patterns—Redux for auth, React Router for navigation, the Notifications API for background alerts.

The real challenge for me was deployment. Up until this point, I'd been able to use managed services to deploy apps. Daily Story, for instance, is a single Node.js app that runs on Cloud Run with no state to manage. But OneTimeChat requires a frontend, backend, and a Postgres database—a setup that demands more than just deploying to a managed platform.

The best way to deploy this was to set up a server myself with production builds of each component.

It was my first time deploying a full-stack application to a self-managed server. I had to think about:

  • Building production versions of both the React app and Node.js backend
  • Setting up and configuring a Postgres database
  • Configuring Nginx as a reverse proxy to route requests to the right services
  • Managing environment variables and secrets across different processes
  • Setting up SSL certificates for secure WebSocket connections
  • Ensuring the Node.js service restarts automatically if it crashes

It was a great experience to step out of the managed world and take more responsibility for how the app runs in production. Understanding the deployment process—from build pipelines to process management to reverse proxies—gave me a much deeper appreciation for what managed services abstract away.

What I Learned

This project was an exercise in using a wide variety of skills. The Notifications API for background alerts, CSS to get a message thread that looked precisely right, Socket.IO for real-time communication, and Redux for state management all came together to create a cohesive experience.

But the most valuable lesson came from deployment. Managed services are powerful, but understanding what happens underneath—how processes are managed, how routing works, how databases connect to applications—makes you a better developer regardless of what platform you ultimately deploy to.
